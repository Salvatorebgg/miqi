# Miqi 学习平台设计规格

日期：2026-07-18  
状态：已确认，待实施

## 1. 产品目标

Miqi 是面向个人长期学习的综合型网站。它以数学主线为核心，将 IELTS 英语、每日资讯、脑力游戏和任务规划整合到一个清晰的“学习驾驶舱”中。首版必须能够在桌面端与移动端使用，通过邮箱密码或 GitHub 登录，并在不同设备间同步学习数据。

成功标准：

- 用户能从高中数学基础循序渐进学习高等数学和线性代数。
- 每个数学章节均包含原理讲解、示例、分层习题、答案解析与外部资源。
- 用户每天能完成词汇、情景理解、英文新闻和论文阅读任务。
- AI、金融、体育、学术等资讯每天自动更新，并允许用户主动刷新最新缓存。
- 数独和迷宫可直接游玩，并记录用时、难度和成绩。
- 首页明确展示今日任务、章节进度、连续学习天数和薄弱项目。
- 网站通过 GitHub Actions 自动测试并部署到 GitHub Pages。

## 2. 明确不做的范围

首版不包含社交动态、排行榜、付费订阅、教师后台、实时聊天或用户生成课程。资讯系统只聚合标题、摘要、来源与原文链接，不复制完整受版权保护的文章。学习建议是辅助规划，不替代正式考试、投资或专业意见。

## 3. 技术方案

采用 React、TypeScript 和 Vite 构建单页应用，使用 GitHub Pages 托管。Supabase 提供邮箱密码登录、GitHub OAuth、PostgreSQL 数据库和跨设备同步。GitHub Actions 负责测试、构建、部署和定时资讯更新。

选择该方案的原因：

- 完全匹配 GitHub 仓库与 GitHub Actions 部署要求。
- 静态前端加载快、成本低，且可安装为 PWA。
- Supabase 的身份认证、关系型数据和行级安全策略适合学习记录。
- 课程、游戏和资讯模块可独立维护，后续扩展不会相互耦合。

## 4. 信息架构

### 4.1 学习驾驶舱

登录后的默认页面采用三栏布局：

- 左侧固定导航：驾驶舱、数学、英语、资讯、脑力、规划、账户。
- 中间主区域：今日最重要的学习内容、快速继续、每日任务与薄弱知识点。
- 右侧上下文栏：今日完成率、最新资讯、脑力休息、本周目标。

移动端将固定侧栏收敛为底部标签栏，次要信息进入可展开卡片。任何页面都保留“继续学习”和同步状态入口。

### 4.2 数学课堂

学习路径分为五个阶段：

1. 高中衔接：数与式、方程不等式、函数、三角、向量与解析几何。
2. 微积分预备：数列、极限直觉、连续性与变化率。
3. 单变量微积分：导数、微分、中值定理、积分与应用、级数。
4. 多变量微积分：空间几何、偏导数、重积分与基础向量分析。
5. 线性代数：线性方程组、矩阵、行列式、向量空间、线性变换、特征值、正交与最小二乘。

每章使用统一课程结构：学习目标、先修知识、直觉说明、正式定义或定理、分步推导、例题、基础练习、进阶练习、答案解析、章节测验、视频与网页资源。完成度由内容阅读、练习正确率和章节测验共同计算。

### 4.3 IELTS 英语

目标能力以 IELTS 8–9 分为方向，不承诺考试分数。功能包括：

- 每日词汇任务与间隔重复，支持认识、模糊、不认识三档反馈。
- 情景理解：校园、工作、旅行、学术讨论等短对话和任务题。
- 英文新闻精读：词汇提示、主旨题、推断题与摘要练习。
- 论文阅读：标题、摘要、研究问题、方法、结果与局限的结构化阅读。
- 周度能力视图：词汇、阅读速度、理解正确率和连续学习天数。

### 4.4 资讯中心

资讯分类至少包括 AI、金融、体育、学术和综合。每条内容保存标题、短摘要、来源、发布时间、分类、原文 URL 和抓取时间。界面支持分类筛选、关键词搜索、已读状态、收藏、刷新和跳转原文。

GitHub Actions 每天四次执行抓取脚本，读取公开 RSS 或无需私密凭证的稳定 API，进行规范化、去重、分类和时间排序，然后生成静态 JSON。抓取部分失败时保留上一份成功数据，记录警告但不阻断网站部署。

### 4.5 脑力游戏

- 数独：至少简单、中等、困难三个难度，支持候选数字、撤销、提示、错误检查和计时。
- 迷宫：可生成多种尺寸，支持键盘、触摸和屏幕按钮，记录步数、用时和最佳成绩。

游戏分数进入账户活动记录，但不强制影响主课程完成度。

### 4.6 规划中心

规划中心显示今日任务、周目标、课程路径、连续学习天数、学习热力图、完成率和薄弱知识点。系统根据未完成任务、练习错误和遗忘曲线生成次日建议；用户可以手动调整任务优先级和每日目标量。

## 5. 视觉与交互

已确认使用“学习驾驶舱”布局，视觉主色为薄荷绿，整体遵循清透的 iOS 风格：

- 冰白到薄荷绿的柔和渐变背景。
- 多层毛玻璃卡片、细白描边、柔和阴影和局部高光。
- 低强度动态水流光晕作为背景氛围，不影响文字阅读。
- 主色以薄荷绿、青绿色和深森林绿建立对比，正文不使用低对比浅色。
- 按钮具备轻微位移、光泽和声效反馈；声效由 Web Audio API 合成，默认音量克制。
- 提供一键静音和“减少动态效果”，并尊重 `prefers-reduced-motion`。
- 键盘焦点、表单错误、加载和同步状态必须清晰可见。

## 6. 身份认证与数据模型

认证方式为邮箱密码与 GitHub OAuth。邮箱注册支持验证和密码重置。GitHub OAuth 回调地址由 Supabase 控制台提供，并加入 GitHub OAuth App 设置。

主要数据实体：

- `profiles`：用户昵称、头像、时区、偏好设置。
- `course_progress`：课程、章节、完成度、测验成绩、更新时间。
- `exercise_attempts`：题目、答案、正确性、耗时和尝试次数。
- `daily_tasks`：日期、类别、目标、完成状态和排序。
- `vocabulary_progress`：单词熟悉度、复习间隔、下次复习时间。
- `reading_attempts`：材料类型、正确率、阅读耗时和摘要。
- `game_sessions`：游戏类型、难度、用时、步数和成绩。
- `saved_articles`：资讯收藏与已读状态。
- `sync_events`：离线队列所需的操作标识、版本和更新时间。

所有用户数据表必须启用 Supabase Row Level Security，并以 `auth.uid()` 限制为当前用户。任何服务端密钥、GitHub OAuth Client Secret 或数据库管理凭证都不得进入前端代码和 Git 仓库。Supabase 公开 URL 与 publishable key 通过构建环境变量注入。

## 7. 本地优先同步

用户操作先写入本地存储并立即更新界面，然后异步同步到 Supabase。离线时操作进入有序队列，恢复连接或下次登录后重试。记录包含稳定 ID、版本号和更新时间；同一字段发生冲突时采用最后更新时间优先，不同字段尽量合并。

同步状态分为：已同步、正在同步、离线待同步和同步失败。失败时保留本地数据并允许手动重试。Supabase 尚未配置时进入明确标识的本地体验模式，所有主要功能仍可使用，但不宣称已实现跨设备同步。

## 8. 数据流与错误处理

课程数据随应用版本发布；用户进度只进入本地存储和 Supabase。资讯抓取脚本将第三方源转换为统一结构，前端只读取已生成的 JSON，不在浏览器中直接抓取跨域站点。

错误处理原则：

- 登录失败显示可操作原因，不泄露账户是否存在等敏感细节。
- 网络失败不清除表单、答案或学习记录。
- 资讯源部分失败继续使用其他来源与上一份成功缓存。
- 外部资源链接失效时隐藏或标记该资源，不影响课程正文。
- 同步冲突和重试有可见状态，但不使用阻塞式弹窗打断学习。
- 全局错误边界提供恢复入口，并记录不含隐私数据的诊断信息。

## 9. 测试与验收

单元测试覆盖：

- 课程进度与连续学习天数计算。
- 练习判分、章节解锁和薄弱知识点聚合。
- 词汇间隔重复调度。
- 资讯规范化、去重、分类和失败回退。
- 离线队列、重试与冲突合并。
- 数独解题校验和迷宫连通性。

浏览器端到端测试覆盖：注册/登录界面、继续学习、答题与解析、每日英语任务、资讯筛选与刷新、数独、迷宫、进度展示、静音与减少动态效果、移动端导航。

构建必须通过 TypeScript 检查、单元测试和生产构建。部署后检查首页、静态资源路径、路由回退、资讯 JSON 和 GitHub Pages 地址。认证流程在 Supabase 与 GitHub OAuth 配置完成后做真实线上验证。

## 10. 部署与运维

仓库使用 `main` 作为默认分支。GitHub Actions 工作流包含：

1. `ci.yml`：安装依赖、类型检查、测试和构建。
2. `pages.yml`：上传生产构建产物并部署 GitHub Pages。
3. `refresh-news.yml`：定时或手动抓取资讯，提交更新后触发部署。

目标线上地址为 `https://salvatorebgg.github.io/miqi/`。Vite 的 base path、OAuth redirect URL 和 PWA scope 均按 `/miqi/` 配置。

部署前需要一次性完成：创建 Supabase 项目、应用数据库迁移、启用邮箱认证、建立 GitHub OAuth App、在 Supabase 设置 OAuth 凭证和重定向白名单，并在 GitHub Actions 配置必要的公开构建变量。生产邮件验证若超出 Supabase 默认邮件服务限制，应配置自有 SMTP。

## 11. 实施边界

本规格作为一个完整首版实施计划执行，但代码必须按模块分层：应用外壳、认证与同步、课程引擎、英语训练、资讯、游戏、规划与设计系统分别维护。课程内容和资讯源属于数据层，不与页面逻辑耦合。所有功能在无账户的本地体验模式可演示；跨设备同步仅在真实 Supabase 配置并验证后标记为可用。
